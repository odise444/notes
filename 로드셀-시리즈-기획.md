## 시리즈 개요

- **시리즈명**: Load-Cell
- **부제**: 로드셀이 거짓말을 한다 — 임베디드 힘 센서 트러블슈팅 연대기
- **폴더**: `content/posts/loadcell/`
- **파일명**: `loadcell-NN.md`
- **이미지**: `static/imgs/loadcell-NN/` 하위, 본문 참조는 `![alt](loadcell-NN/01.png)`
- **태그 후보**: 로드셀, P7001, 무게측정, UART, STM32, CANopen, 트러블슈팅, 프로토콜분석, 안전기능
- **원본 초안**: `blog_loadcell_troubleshooting_draft.md` (5막 구성, 약 2주 디버깅 기록)

## 익명화 원칙

- **실제 프로젝트 언급 금지**: 프로젝트명, 장비 용도의 구체적 식별 정보, 보드 실크·로고·회사 관련 정보 일절 노출 금지. 용도는 "사용자 조작력을 측정하는 안전 기능 입력" 수준의 일반 서술까지만.
- **모듈 정보는 공개**: P7001-2 모델명, 프로토콜 바이트, 미문서 ADC raw 명령(B0 34 E4) 모두 공개 가능. 모듈 자체에 대한 리버스엔지니어링·실측 내용은 제한 없음.

## 시스템 구성 (공통 배경, 각 편 도입부에 요약)

- 로드셀 ×2 + P7001-2 무게 변환 모듈 (UART 9600 8N1, 폴링)
- STM32F103 인터페이스 보드, LC1=USART1 / LC2=USART2
- CAN(CANopen TPDO) → 리눅스 마스터(Python)
- 힘 데이터는 과부하 E-STOP 등 안전 기능 입력 → 품질 문제가 아니라 안전 문제

## 포스트 계획

| #   | 제목 (가제)                                          | 핵심 내용 (원본 초안 매핑)                                                                                          | 상태 |
| --- | ---------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- | ---- |
| 1   | 매뉴얼이 틀렸다, 그리고 89ms의 진범                  | 1막(read 에코 0x31 vs 0x33, raw 덤프로 파서 교정) + 2막(89≈11+10+50+10 타이밍 산수, 블로킹 폴링 → 비블로킹 상태머신) | ✅ 완료 |
| 2   | 숫자가 멈췄다 — 살아있는 척하는 모듈                 | 3막. 동적 부하 시 무게값 동결, CAN→펌웨어→직결 레이어 격리, 미문서 ADC raw 명령(B0 34 E4) 발견, 자체 환산으로 필터 우회 |      |
| 3   | 깨진 바이트 스트림 — RS-232를 TTL에 직결하면         | 4막. LC2 오정렬 46%, 트랜시버 미경유 직결, straddle 증폭, 헤더 헌트 + FLUSH 상태로 오정렬 0% (무응답은 HW 잔존)      |      |
| 4   | 안전 계층과 캘리브레이션의 정착                      | 5막. FORCE_STALE E-STOP 도입과 비활성 결정(오탐 vs 미감지 trade-off), SDO tare/span flash 영속화, 원자성 3원칙       |      |

분할 근거: 2편(3막)이 단독으로 가장 강한 글감이라 독립 배치. 1막+2막은 가벼운 디버깅 스토리로 도입, 4막·5막은 분량상 각각 분리 (straddle 메커니즘과 safety trade-off 모두 다이어그램·표가 필요해 합치면 과밀).

## 첨부 자료 (편별)

- 1편: 시스템 구성도(익명화), raw 덤프 캡처, 89ms 타이밍 분해 다이어그램
- 2편: 동결 구간 ADC raw vs 무게 동시 플롯 (결정적 증거), 레이어 격리 순서도
- 3편: LC1/LC2 60초 폴 통계 막대그래프, straddle 타임라인 다이어그램
- 4편: FORCE_STALE 트립 타이밍 측정, 캘리 커밋 시퀀스

## 공개 전 점검 (매 편 필수)

- [ ] 플롯/캡처/회로도/사진에 프로젝트명·보드 실크·로고·장비 식별 정보 노출 여부
- [ ] 본문에 실제 프로젝트를 특정할 수 있는 서술 없는지 확인

## 작성 규칙

- 각 포스트 완료 시 이 문서의 상태 컬럼을 ✅ 완료로 갱신하고 섹션 요약 추가
- 톤: 직설적 `~다` 체, 이모지/별점/과한 볼드 금지, 실측 데이터 근거
- 각 편 끝에 해당 막의 교훈만 정리, 전체 교훈 요약(9개)은 마지막 편 마무리에 배치

## 완료 포스트 요약

### #1 매뉴얼이 틀렸다, 그리고 89ms의 진범 (2026-06-12, loadcell-01.md)

- 들어가며: 시리즈 배경(안전 기능 입력이라 끝까지 판 이유), 시스템 구성, P7001-2 프로토콜 소개
- 1막: garbage 값 → raw 덤프 → 에코 0x31(매뉴얼) vs 0x33(실물), 오프셋 어긋남 → 파서 교정 코드, 583.8g/5.72N 검산
- 2막: 89ms 갱신 → 타이밍 산수(11+10+50+10) → 미연결 LC2의 50ms 타임아웃이 진범 → 채널별 비블로킹 상태머신 코드 → 10~20ms로 개선
- 마무리: 교훈 3개, 2편(동결) 예고
