## 익명화 원칙

- **실제 프로젝트 언급 금지**: 프로젝트명, 장비 용도의 구체적 식별 정보, 보드 실크·로고·회사 관련 정보 일절 노출 금지. 용도는 "사용자 조작력을 측정하는 안전 기능 입력" 수준의 일반 서술까지만.
- **모듈 정보는 공개**: P7001-2 모델명, 프로토콜 바이트, 미문서 ADC raw 명령(B0 34 E4) 모두 공개 가능. 모듈 자체에 대한 리버스엔지니어링·실측 내용은 제한 없음.

## 프로토콜 실측 메모 (본문 작성 시 참조)

- 무게 읽기: `B0 31 E1` → 8B 응답 `B0 [에코 0x33] [unit] [무게×100 BE signed ×4] [cksum]`
- ADC raw: `B0 34 E4` (미문서) → 6B 응답 `B0 [raw BE unsigned ×4] [cksum]` — 에코/unit 없음
- ADC는 24bit 미드스케일 기준: counts = raw − 0x800000 (무부하 실측 +19,030)
- 체크섬 공통: 직전 바이트 합의 하위 8bit

## 시스템 구성 (공통 배경, 각 편 도입부에 요약)

- 로드셀 ×2 (빔타입, 정격 100kg, 1.9831mV/V) + P7001-2 무게 변환 모듈 (UART 9600 8N1, 폴링)
- STM32F103 인터페이스 보드, LC1=USART1 / LC2=USART2
- CAN(CANopen TPDO) → 리눅스 마스터(Python)
- 힘 데이터는 과부하 E-STOP 등 안전 기능 입력 → 품질 문제가 아니라 안전 문제

## 첨부 자료 (편별)

- 1편: 모듈 보드 사진, 제조사 유틸, 자작 툴, raw 덤프 캡처 — ✅ 4장 반영
- 2편: 로드셀/지그 사진 2장 ✅ 반영. 동결 구간 ADC raw vs 무게 동시 플롯 — **미반영, 자료 확보 시 추가**
- 3편: 폴 통계 가로 누적 막대, straddle 개선 전/후 타임라인 — ✅ 2장 반영
- 4편: FORCE_STALE 트립 타이밍 측정, 캘리 커밋 시퀀스

## 공개 전 점검 (매 편 필수)

- [ ] 플롯/캡처/회로도/사진에 프로젝트명·보드 실크·로고·장비 식별 정보 노출 여부
- [ ] 본문에 실제 프로젝트를 특정할 수 있는 서술 없는지 확인

## 작성 규칙

- 각 포스트 완료 시 이 문서의 상태 컬럼을 ✅ 완료로 갱신하고 섹션 요약 추가
- 톤: 직설적 `~다` 체, 이모지/별점/과한 볼드 금지, 실측 데이터 근거
- 각 편 끝에 해당 막의 교훈만 정리, 전체 교훈 요약은 마지막 편 마무리에 배치

## 완료 포스트 요약

### #1 매뉴얼이 틀렸다, 그리고 89ms의 진범 (2026-06-12, loadcell-01.md)

- 들어가며: 시리즈 배경(안전 기능 입력이라 끝까지 판 이유), 시스템 구성, P7001-2 프로토콜 소개
- 1막: garbage 값 → raw 덤프 → 에코 0x31(매뉴얼) vs 0x33(실물), 오프셋 어긋남 → 파서 교정 코드, 583.8g/5.72N 검산
- 2막: 89ms 갱신 → 타이밍 산수(11+10+50+10) → 미연결 LC2의 50ms 타임아웃이 진범 → 채널별 비블로킹 상태머신 코드 → 10~20ms로 개선
- 이미지 4장: raw 덤프(01), 자작 툴(02), 모듈 보드(03), 제조사 유틸(04)
- 마무리: 교훈 3개, 2편(동결) 예고

### #2 숫자가 멈췄다, 살아있는 척하는 모듈 (2026-06-12, loadcell-02.md)

- 증상: 동적 부하 시 force 동결, 안전 기능 입력이라 치명적. 벤치 구성(100kg 빔타입 로드셀+지그) 소개
- 레이어 격리: CAN 캡처(TPDO 12Hz 정상, 폴 성공 +32회인데 force bit-exact 동일) → 펌웨어 리뷰(드레인+체크섬, stale-read 아님) → 시리얼 직결(12.30g×139프레임 14초 반복 후 자가 회복, 모듈 측 확정)
- "살아있는 척" 분석: valid 프레임 ≠ fresh 데이터, heartbeat 감시 무력화, 안전 우회 구조
- 미문서 명령 B0 34 E4 발견: 응답이 read 골격과 다른 6B 비대칭(에코/unit 없음), 24bit 미드스케일(−0x800000) 구조 분석, 실측 프레임 대조 + 전용 파서 코드
- ADC/무게 동시 캡처로 동결이 무게-변환(모션 필터) 단계만임을 확증 (ADC 즉시 추종, 선형성 70.6 counts/g)
- 해결: ADC raw + 2점 선형 자체 환산으로 필터 우회 (미드스케일 오프셋은 2점에서 상쇄), 동결 재현 조건에서 라이브 추종 검증
- 이미지 2장: 로드셀 상면(01), 직결 벤치(02). ADC vs 무게 동시 플롯은 자료 확보 시 추가
- 마무리: 교훈 4개, 3편(프레임 깨짐) 예고

### #3 깨진 바이트 스트림, RS-232를 TTL에 직결하면 (2026-06-12, loadcell-03.md)

- 증상: LC2만 폴 실패 버스트(최대 1.2초). 60초 통계 LC1 81/2/17 vs LC2 22/46/32로 정량화
- 근본 원인: LC2가 트랜시버 없이 RS-232 출력을 3.3V TTL 직결 (블루투스 커넥터 재활용), 마진 클램핑으로 간헐 비트 깨짐
- 증폭 메커니즘: 지각 프레임(6B, 타임아웃 90ms) → straddle → 헤더 0xB0 밀림 → 600ms+ 연속 실패 버스트
- 해결: 헤더 헌트(0xB0 동기, 체크섬 2차 필터) + FLUSH 상태(정숙 3ms/상한 20ms) → raw 오정렬 4.4% → 0.0%
- 무응답은 HW 잔존 명시, 트랜시버 경유 배선이 진짜 수리 (보드 개정 등재). 펌웨어 완화 ≠ HW 수리 구분 기록
- 이미지 2장: 폴 통계 가로 누적 막대(01), straddle 개선 전/후 타임라인(02)
- 마무리: 교훈 3개, 4편(안전 계층/캘리 영속화) 예고
